# Agent Guidelines

This document outlines the coding standards, tech stack, and project structure for the **NFC Mock-up** project. AI agents should refer to this file to ensure consistency and quality.

## 1. Tech Stack

- **Runtime**: Browser (Vite)
- **Framework**: React 19
- **Language**: TypeScript (Strict Mode)
- **Build Tool**: Vite 7
- **Package Manager**: Yarn (Berry)
- **Routing**: React Router DOM v7

### UI & Styling
- **CSS Framework**: Tailwind CSS v4
- **Component Primitives**: Radix UI (`@radix-ui/*`), Vaul (Drawer)
- **Icons**: Lucide React
- **Animations**: Framer Motion v12
- **Notifications**: Sonner
- **Utility Libraries**: `clsx`, `tailwind-merge`, `class-variance-authority` (cva)

## 2. Project Structure

- **`src/`**: Source code root (aliased as `@/`)
  - **`components/`**: Reusable UI components (buttons, drawers, etc.)
  - **`components/ui/`**: Shadcn/UI-like base components
  - **`pages/`** or **`scan/`**: Feature-based directories (e.g., `src/scan/page.tsx`)
  - **`types/`**: TypeScript type definitions
  - **`context/`**: React Context definitions
- **`public/`**: Static assets
- **`vite.config.ts`**: Vite configuration

## 3. Coding Standards & Conventions

### TypeScript & React
- **Functional Components**: Always use functional components with Hooks.
- **Strict Typing**: Avoid `any`. Define explicit interfaces or types for props and state.
- **Component Granularity**:
  - **Extract Large Components**: If a component file grows too large (e.g., > 200 lines) or contains distinct UI sections (TopBar, Drawers, Modals), **extract them into separate files**.
  - **File Placement**: Place feature-specific sub-components in a `components/` folder within the feature directory (e.g., `src/scan/components/TopBar.tsx`).
- **Naming**:
  - Components: `PascalCase` (e.g., `ScanPage.tsx`, `Button.tsx`)
  - Functions/Variables: `camelCase`
  - Constants: `UPPER_SNAKE_CASE`
- **Imports**: Use the `@/` alias for local imports to keep paths clean.
  ```typescript
  import { Button } from "@/components/ui/button"; // Correct
  import { Button } from "../../components/ui/button"; // Avoid
  ```

### Styling (Tailwind CSS)
- **Utility First**: Write styles using Tailwind utility classes directly in `className`.
- **Dynamic Classes**: Use `cn()` (clsx + tailwind-merge) for conditional class names.
  ```typescript
  <div className={cn("bg-white", isActive && "bg-blue-500")} />
  ```
- **Animations**: Use `framer-motion` for complex interactions and `tailwindcss-animate` (or standard Tailwind utilties) for simple transitions.

### State Management
- Use local state (`useState`) for component-specific logic.
- Use Context API for global or feature-wide state (e.g., `TicketContext`).
- Avoid Redux unless complex global state is required.

### Best Practices
- **Clean Code**: Keep components small and focused. Extract logic into custom hooks if a component grows too large.
- **Accessibility**: Use semantic HTML and ensure interactive elements have proper ARIA attributes (Radix UI handles much of this).
- **Mocking**: For mockups, keep mock data typed and separated or clear within the component (as seen in `ScanPage`).

### Dashboard Aesthetic (Luma Style)
- **Header Navigation**: Use a minimalist top header (`h-[60px]`, `text-md`) instead of a sidebar.
- **Flat Z-Index**: All components should effectively sit at `z-0` or similar flat hierarchy unless absolutely necessary (modals).
- **Typography**: Minimalist sans-serif (Inter/system-ui).
- **Actions**: Login/Profile buttons should be subtle (`text-gray-400`, `hover:text-white`).

## 4. Workflow
- **Linting**: Ensure code passes ESLint (`yarn lint`).
- **Formatting**: Maintain consistent formatting (Prettier/EditorConfig recommended).
- **Git**: Commit messages should be clear and descriptive.

When modifying or creating code:
1. **Check Existing Patterns**: Look at `src/scan/page.tsx` or similar files for reference.
2. **Preserve UI/UX**: This is a mock-up/prototype; visual fidelity and smooth animations (Framer Motion) are high priority.
